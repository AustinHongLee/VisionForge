## 零、換電腦快速規則（給李宗鴻）

你在這個專案的預設角色是**快遞＋蓋章**，不是技術判斷者。換到另一台電腦時，先做：

```powershell
git clone https://github.com/AustinHongLee/VisionForge.git
cd VisionForge
git status --short --branch
```

接著把本手冊、`docs/README.md`、最新規格票貼給當前 Builder／Architect。除非你主動想看實作，否則只做三件事：

1. 跑 Architect 給你的 git 指令。
2. 把 Architect 發出的規格票原文轉貼給 Codex／Builder。
3. Codex 回報 L0 票 CI 綠時，跑 Architect 或 Codex 給的合併指令。

任何「要不要這樣設計」、「這個狀態是否正確」、「某票能不能合併」都不該丟給你判斷。Architect 自己查，Builder 自己施工，CI 自己守門。

4. 回報「已就緒」＋你認為的下一步，等決策者（李宗鴻）一句話確認後動工。

## 四、環境怪癖（血淚清單，全部真實發生過）

1. **沙箱 git 不可用**（掛載層不支援 unlink／lock，會寫壞 config）→ git 一律由使用者或 Builder 在 Windows 端執行，你只給複製貼上的指令區塊。
2. **掛載同步會截斷檔案**（已發生 3 次：測試檔尾端、`contracts/__init__.py`、`pyproject.toml`——都斷在中間）→ 關鍵設定與程式檔用 **bash heredoc 直寫**；任何 Write／Edit 後用 bash 驗證（`tail`／`wc`／`ast.parse`）；**每輪結尾跑完整測試**。
3. 沙箱 Python 3.10 vs 專案目標 3.12 → 程式碼保持 3.10 可執行（ruff 已 ignore UP017／UP042，理由在 pyproject 註解）；別讓 ruff --fix 把它們改回 3.11+ 語法。
4. pip 需 `--break-system-packages`；pytest 前清 `__pycache__` 並用 `-B -p no:cacheprovider`（掛載 mtime 不可靠會吃到舊 bytecode）。
5. 使用者主控台為 cp950，git 輸出中文會亂碼——屬顯示問題，commit 本體正常，別誤判。
6. Windows 端可能有未提交工作樹變更；提交時一律 `git add` 指定檔案，不用 `git add -A`，除非 Architect 明確給出該指令並承擔範圍。

## 五、工作慣例（使用者已習慣，不要改）

1. **使用者不入場原則**：他只做兩件事——跑你給的指令、轉貼規格票給 Codex。不要請他讀內容或做技術判斷；介面出來前他不操作產品。
2. **每輪結尾必附「📋 下一步表」**：誰／做什麼／怎麼做，三欄。
3. 語言繁中、力求簡潔；commit 格式 `type(scope): 中文描述`。
4. **L1 抽審計數**：L0 已合併 1 張（票-0002，抽查通過）；下次抽查在第 3 張 L0 合併後。
5. ADR 採認模式：使用者已授權「**合併即視同採納**」（有異議他會說）；仍須在 ADR 狀態欄註明。
6. 治理三義務：新文件登記 `docs/README`、新名詞登記定義表、狀態變更同步——你自己也要遵守。
7. Builder（Codex）目前信任紀錄：兩張票零瑕疵；規格票缺欄它會退票，這是預期行為。

## 六、標準工作流

### Architect → 使用者

Architect 負責 ADR、規格票、合併策略與審查層級判斷。每輪結尾必給：

```text
誰／做什麼／怎麼做
```

如果需要 git 操作，Architect 必須給完整 PowerShell 指令，且列出要 stage 的檔案。不要要求使用者自行判斷「哪些檔案該進 commit」。

### 使用者 → Codex／Builder

使用者只轉貼票面：

```text
施工票-NNNN，票面在 docs/06-規格票/票-NNNN-標題.md。
分支開 codex/ticket-NNNN-短名。
只准動票面白名單。
```

Builder 施工完成後回報 commit、驗證、剩餘風險。L0 票 CI 綠即可依規則合併；L1/L2 依 Architect 指令。

### Builder → GitHub

無 remote 時先設定：

```powershell
git remote add origin https://github.com/AustinHongLee/VisionForge.git
git push -u origin main
git push -u origin codex/ticket-NNNN-短名
```

已有 remote 時只推當前需要的分支。未提交工作樹變更不會被 push，但會影響切分支與合併，操作前必先 `git status --short --branch`。

## 七、給接任模型的運行建議

| 工作類型 | 建議設定 |
|---|---|
| Kernel 設計（契約、統計、架構取捨） | Extended thinking 開高（預算從寬） |
| L2 審查、發票、ADR、治理 | 標準模式即可 |
| R2 等級的大型評審、修憲 | 最高 thinking 預算；必要時建議決策者用更高階模型做一次性檢視 |

暖機建議：就任後前一兩輪先做低風險工作（發票-0003、處理登記表），確認慣例吻合後再碰 kernel（校準引擎）。
