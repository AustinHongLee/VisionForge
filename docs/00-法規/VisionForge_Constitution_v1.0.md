## 前言

本憲法回答一個問題：**「這個產品永遠不能違反什麼？」**

它不規定現在該做什麼（那是規劃書的事），不規定怎麼實作（那是 ADR 與程式碼的事）。它只規定不可跨越的邊界。條文越少、越硬，憲法越有效；因此本文件刻意精簡，凡可下放到規劃層的內容一律不寫。

**規範用語**（全文一致）：

- **必須（MUST）**：無條件遵守，違反即違憲。
- **不得（MUST NOT）**：無條件禁止，違反即違憲。
- **應（SHOULD）**：預設遵守；偏離時必須以 ADR 記錄理由。
- **可（MAY）**：允許，不強制。

**修憲程序**：本憲法的任何修改必須：升版（v1.0 → v1.1 → v2.0）、附書面理由與影響分析（ADR 形式）、逐條標注變更。開發過程中若發現憲法阻礙了正確的設計，唯二合法出路是**修憲**或**修改設計**；**繞過憲法即違憲**，不存在第三條路。

**憲法自身的邊界**：本憲法不指定程式語言、框架、資料庫產品、UI 形態——這些屬於 ADR 層級，正因為它們必然改變，才不能入憲。

**關於「Vision AI Operating System」一詞**：本憲法採用 OS 作為**內部架構隱喻**——核心（Orchestrator 與治理）如同 kernel，Provider 如同驅動程式，工作流與應用管線如同 userland，資產如同檔案系統。此隱喻用於約束依賴方向與職責邊界。**不得**以「OS」的宏大敘事正當化任何範圍擴張：本產品在使用者眼中永遠是一個開箱即用的桌面工具，隱喻是給架構師的，不是給行銷的。

# 第一章　產品第一性原理（First Principles）

**P1（販售之物）** 本產品販售的不是模型、不是標註工具、不是訓練功能。本產品販售兩樣東西：**被驗證的視覺能力**（使用者確知它有多可靠，因為有證據）與**會增值的視覺資產**（資料、標準、記憶——隨使用時間累積價值，且不隨任何模型的死亡而貶值）。一切功能取捨以是否服務這兩樣東西為判準。

**P2（資產與耗材）** 模型是耗材，資料是資產。Dataset、Golden Dataset、Taxonomy、血統帳本、Project Memory 是使用者的財產；任何模型（不論老師或學生）都是可拋棄、可替換的加工設備。**當模型的便利與資料的完整性衝突時，資料必勝，無例外。**

**P3（必變性假設）** 一切具名模型必亡。GPT、Claude、Gemini、Grounding DINO、SAM、YOLO、RT-DETR——本憲法生效十年內，以上每一個都會被取代或面目全非。因此**核心架構不得依賴任何具名模型的存在**；為替換而設計，不為任何模型而設計。

**P4（人的注意力是最稀缺資源）** 人的判斷是系統中唯一不可再生的資源。**不得**讓人做機器可靠可做的事（例如從零手繪機器能畫的框）；**必須**讓每一次人為判斷都被永久捕捉為資產（審核紀錄、記憶條目、黃金樣本）。花掉人的一分鐘而未留下任何可累積的痕跡，即為系統設計失敗。

**P5（可驗證性）** 不可驗證的能力視同不存在。任何「模型很準」「資料很乾淨」「這批可以入庫」的宣稱，必須存在對應的證據路徑（可點開、可複核、可重放）。信任建立在證據上，不建立在品牌、參數量或信心分數上。

**P6（使用者主權）** 使用者的影像、標註、記憶、帳本屬於使用者。**必須**：專案為自包含、格式開放文件化、離線可用核心功能、可整包帶走。**不得**：未經明示同意將任何使用者資料送離本機、以任何專有格式造成鎖定（包括被本產品自己鎖定）。

**P7（十年常數與變數）** 設計投資必須押在常數上。常數：人類需要機器看懂「自己的東西」、人類看不懂原始指標、信任需要驗證、資料品質決定一切下游、人的注意力稀缺。變數：模型、Provider、價格、任務邊界、模態、介面形態。**凡建立在變數上的設計，必須以可拋棄的方式建造。**

**P8（複雜度的歸屬）** 複雜度是核心的負擔，簡單是使用者的權利。Orchestrator、Provider、路由、校準——這些概念存在的目的是讓使用者**不需要知道它們存在**。預設隱形、隨時可查（透明可稽核，但絕不強迫理解）。若使用者被迫理解內部架構才能完成工作，即為違憲的洩漏。

# 第二章　Architecture Principles

**A1（Orchestrator 至上）** Vision Orchestrator 高於一切 Provider。任何 Provider——不論多強、多便宜、多流行——都只是 Orchestrator 調度的工具。**YOLO 永遠不能是核心；任何模型永遠不能是核心。** 核心是調度、治理與資產管理。

**A2（Provider 可替換性）** 任何 Provider 必須可在不修改核心程式的前提下新增、移除、升級、降級（詳第五章）。核心程式碼**不得** import 任何 Provider 的 SDK；此規則必須以自動化檢查（依賴規則檢測）強制執行。

**A3（依賴方向單向）** 依賴方向固定為：介面層 → 服務層 → Orchestrator → Provider。反向依賴一律禁止。核心不依賴任何 GUI 框架、任何具體任務類型、任何具體 Provider。核心必須可完全無頭（headless）運行——GUI 只是核心的其中一個客戶端。

**A4（Dataset 高於 Model）** 資料流的完整性優先於模型流的便利性。任何為了「讓訓練跑起來」「讓某 Provider 好接」而破壞血統、跳過版本、繞過閘門的設計，一律違憲。

**A5（確定性內核、機率性外圍）** 一切記帳、路由紀錄、版本管理、閘門裁決邏輯必須是確定性的。機率性元件（一切模型，含 LLM）被隔離在介面之後，其輸出被記錄為資料。**LLM 不得出現在記帳、裁決、版本控制的路徑上**——它可以建議，不可以記帳。

**A6（證據原則）** 沒有證據的事等於沒發生。每一個自動化決策必須攜帶：依據（政策版本、輸入參照）、產出、出處。詳見 D 章各條。

**A7（開放格式）** 專案在磁碟上的形態必須不依賴本產品即可讀取（標準檔案格式＋文件化的結構）。本產品消失後，使用者的資產必須還活著。

**A8（本地優先）** 雲端是加速器，不是依賴。核心工作流（審核、資產管理、本地 Provider 推論）必須在完全離線環境全功能運作。任何「沒有網路就打不開產品」的設計違憲。

**A9（任務不可知）** 核心只認識抽象的「輸入媒體 → 結構化 Claim」。Detect、Segment、Pose、OCR、未來未知任務，一律以任務模組形式存在。核心資料結構**不得**硬編碼任何特定任務的幾何假設（詳 F1）。

**A10（反過度工程）** 任何抽象介面必須有至少兩個真實實作才得以固化（Rule of Two）；只有一個實作的抽象必須標記為暫定。禁止投機性泛化——為想像中的需求預建的通用機制，本身就是技術債。**接口可以預留（一行契約），實作不得預建（一座工廠）。**

**A11（失敗為常態）** 每一個 Provider 呼叫都可能失敗、變慢、變貴、被停服。系統的預設姿態是降級與備援，**不得**存在「假設某 Provider 永遠可用」的路徑。

**A12（人為最終權威）** 任何自動決策，人皆可覆寫；覆寫必勝且被記錄（成為 Project Memory 的養分）。系統可以比人聰明，不可以比人大。

# 第三章　核心名詞定義

本章定義具有憲法效力：全產品的文件、程式、介面文案**必須**使用以下語義，不得私自重定義。

**Claim（斷言）**：一個 Provider 對一份輸入回報的結構化陳述（幾何＋概念＋信心＋完整出處）。Claim 是**有出處的意見，不是事實**。合成／集成的結果（多 Provider 表決產物）仍是 Claim，攜帶複合出處。

**Label（標註）**：經 Human Review 批准（含修正後批准）的 Claim。**只有 Label 能進入 Dataset；Claim 永遠不能直接進入。** 這是全系統最硬的一條語義規則。

**Vision Provider**：任何能接受影像（及可選提示）並回傳 Claim 的元件——本地或雲端、可訓練或不可、單模態或多模態，一視同仁。OpenAI、Claude、Gemini、Grounding DINO、SAM、YOLO、RT-DETR 與一切未來模型，在本架構中的身分完全相同：Provider。

**Provider Capability（能力聲明）**：Provider 以機器可讀形式自我聲明的合約：支援任務、提示模式（文字/框/點/範例圖）、地域（本地/雲端）、成本模型、決定性、可訓練性、輸出型別。聲明是履歷，**試鏡（以 Golden Dataset 實測）才是面試**；路由資格以實測為準。

**Teacher（老師）／Student（學生）**：**角色，不是身分。** Teacher = 該次調用中負責產生 Claim、診斷或評審的 Provider（通常免訓練、貴、不宜部署）。Student = 被訓練以供部署的 Provider（快、小、便宜）。同一 Provider 可在不同調用中扮演不同角色。

**Vision Orchestrator**：唯一有權決定「哪個 Provider 在何時以何成本處理什麼」的核心元件。**它自己永不解讀像素**（詳第四章）。

**Confidence（信心值）**：Provider 自報的原始分數。**原始信心值只在同一 Provider 同一版本內具有排序意義；跨 Provider 比較原始信心值屬於違憲操作。** 必須經 Calibration（以 Golden Dataset 將信心映射至實測精確率）後方可驅動自動分流；**未經校準（或校準信賴度不足）時，一律保守路由至人工審核。**

**Decision（決策）**：Orchestrator 產生的一筆不可變紀錄：時間、輸入參照、候選 Provider 集、生效政策版本、選擇與理由碼、成本預估、實際結果、後續處置。Decision 必須可重放（Replay）。

**Evidence（證據）**：使一筆 Decision 或 Claim 可被人類複核的最小材料集：輸入參照、輸出、依據數據、政策版本。

**Dataset（資料集）**：版本化資產 = 內容雜湊定址的媒體 ＋ Label（含血統）＋ 切分 ＋ Taxonomy ＋ 版本 manifest。一個沒有版本與血統的圖片資料夾**不是** Dataset。

**Golden Dataset（黃金集）**：純人工建立與維護的基準資產。三鐵律：**永不參與任何訓練、永不被任何機器覆寫、是一切裁決的最終依據。** 四職能：仲裁（模型與資料是否變好）、校準（信心→精確率）、試鏡（Provider 在本專案的實測資格）、漂移基準。

**Human Review（人工審核）**：將 Claim 轉為 Label 的**唯一合法程序**。每一次審核行為（誰、何時、改了什麼）本身即是資料，自動沉澱為 Project Memory。

**Project Memory（專案記憶）**：專案內長期累積的結構化事實：詞彙語意（本專案的「Valve」指氣動閥）、修正統計（累計 800 次修正的型態分布）、各 Provider 分類別實測成績、混淆對、校準曲線、成本歷史、生效門檻。屬於專案、存於專案、隨專案攜帶（詳第七章）。

**Knowledge Base（知識庫）**：產品層通用知識（術語字典、方法論、症狀→藥方 playbook）。對專案唯讀，**不含任何使用者資料**。與 Project Memory 的分界：Knowledge Base 是「世界的常識」，Project Memory 是「你的專案的事實」。

**Application Pipeline（應用管線）**：消費 Label／Claim 以產生業務產物（計數、紀錄、警報、報表）的下游流程。屬於 userland；核心對它只承諾一件事：穩定的、版本化的輸出契約。

**Cost Engine（成本引擎）**：計量、預測、預算執行的確定性子系統。**它量測與攔阻，不做選擇**——選擇是 Orchestrator 的職權（詳第六章）。

**Budget（預算）**：使用者或政策設定的資源上限（金錢／時間／調用數），為硬約束。

**Escalation（升級）**：Orchestrator 將工作移交更強 Provider 或人工的行為。**人工是升級鏈的最上層，也是最貴的 Provider。**

# 第四章　Vision Orchestrator

## 4.1 存在理由

Provider 會有很多個、會一直換、會失敗、會漲價。若沒有一個唯一的調度權威，每個功能都會各自硬連自己喜歡的模型，三年後產品變成無法維護的模型義大利麵。Orchestrator 把「用哪個模型」從一百個散落的 if 句，收斂為一個有帳本、有政策、可重放的決策中心。

## 4.2 專屬職責（只有 Orchestrator 可以做，別處出現即違憲）

**O1（Provider Routing）** 依能力聲明、試鏡成績、Project Memory 先驗與生效政策，選擇處理者。

**O2（Confidence Routing）** 依**校準後**信心將 Claim 分流至對應的審核層級（快速確認／逐張審核／人工重標）。注意其輸出是「審核佇列的分派」——**Orchestrator 無權將任何 Claim 直接變成 Label（越過 Human Review 即違憲）。**

**O3（Cost-aware Planning）** 在 Cost Engine 提供的計量、預測與預算約束下規劃調用（先便宜後昂貴、批次合併、快取優先）。

**O4（Fallback 與 Retry）** Provider 失敗時的替補與有界重試（次數與退避由政策規定），全程留痕。

**O5（Human Escalation）** 觸發移交人工的判斷。合法觸發條件（封閉清單，擴充須修訂政策）：校準信心低於類別門檻、多 Provider 分歧超出容忍、預算耗盡、輸入判定為新奇分布、政策指定（如安全關鍵類別）、稽核失敗率突升、使用者主動要求。

**O6（Provider Voting）** 政策要求時徵集多 Provider 意見。表決結果的用途限於**分流**（一致→快速佇列、分歧→人審）與產生複合 Claim；**表決不得產生 Label。**

**O7（Decision Ledger）** 維護不可變、可重放的決策帳本。

**O8（Memory 的首要讀寫者）** 消費 Project Memory 作為路由先驗；將決策結果回寫為記憶。

**O9（Knowledge Cache）** 相同（輸入雜湊、Provider 版本、參數）的 Claim 必須重用而非重呼叫；失效條件為三元組任一變動。

## 4.3 明確不負責（出現在 Orchestrator 內即違憲）

- **不解讀像素**：它永不產生 Claim；它是法官，不是證人。
- **不批准 Label**：批准權專屬 Human Review 程序。
- **不擁有 UI**：無頭運行，GUI／CLI／未來的 Agent 都只是它的客戶端。
- **不執行訓練**：訓練是 Student Provider 的動詞，Orchestrator 只排程與記帳。
- **不做業務邏輯**：Application Pipeline 在它之上。
- **不制定政策**：政策（預算、優先序、隱私開關）由使用者與專案設定制定；Orchestrator 是執行者，不是立法者。

## 4.4 不變量

每一筆 Orchestrator 決策必須同時滿足：源自可指認的政策版本、附帶 Evidence、經 Cost Engine 計量、可重放、可被人覆寫（覆寫亦入帳）。五者缺一即違憲。

**降級階梯（O10）**：無網路 → 僅本地 Provider；無任何 Provider → 純手動模式（審核與資產管理全功能）。Orchestrator 只能降級，**不得**成為阻斷點。

# 第五章　Provider Architecture

## 5.1 目標

新增任何 Provider——包括 2032 年才會發明的——**不修改、不重新編譯核心**。

## 5.2 條文

**PR1（統一契約）** 所有 Provider 實作同一組小動詞集：`describe`（回報能力聲明）、`claim`（輸入媒體＋提示 → Claim）、可選 `train`、可選 `export`、可選 `calibrate` 掛鉤。核心只認識這組動詞。

**PR2（唯一耦合點）** 核心與 Provider 的唯一共享知識是**正規化 Claim Schema** 與能力聲明詞彙表。此 Schema 由核心擁有、語意化版本管理；核心必須同時支援當前版與前一版（N 與 N−1）。

**PR3（發現式註冊）** Provider 以套件形式投放（manifest：身分、版本、相容 API 版本、能力聲明、權限需求），由核心**發現**而非被核心**認識**。核心程式碼中出現任何 Provider 的名字，即違反 A2。

**PR4（程序隔離）** Provider 在核心程序之外執行（獨立 Worker 或沙箱）。Provider 崩潰不得波及核心；Provider 的檔案與網路存取受 manifest 權限約束。

**PR5（試鏡制）** 任何 Provider 在進入自動路由前，必須通過兩關：契約一致性測試（Conformance Suite，驗證動詞與 Schema 行為）與本專案 Golden Dataset 試鏡（實測分類別成績）。未過試鏡的 Provider 只能被使用者手動、顯式地調用。**聲明不可信，實測才可信。**

**PR6（版本即身分）** Provider 升版 = 新的 Provider 實體。血統、校準曲線、記憶條目、快取全部綁定至（Provider, 版本）二元組；升版後校準與試鏡必須重做。**任何 Claim 必須可回答：「你是誰的哪一版、用什麼提示產生的？」**

**PR7（輸出正規化與校準層）** 各 Provider 的座標系、幾何鬆緊、信心語意各不相同。正規化（翻譯為統一 Claim）與校準（信心→實測精確率映射）是 Provider 接入的必要部分，**不得**以「先接上再說」跳過。

**PR8（不得互相呼叫）** Provider 之間不存在直接通道；一切組合（老師教學生、雙老師交叉檢核）必經 Orchestrator。

**PR9（詞彙表只增不改）** 能力聲明詞彙表 append-only：新能力（新模態、新提示型態）以新增詞條容納，**不得**修改既有詞條語意——否則舊 Provider 的聲明將靜默失真。

# 第六章　Cost Strategy（Cost Engine）

## 6.1 權責分立

三權分立，不得合併：**使用者／政策**制定預算與優先序（立法）；**Orchestrator** 在約束內做選擇（行政）；**Cost Engine** 計量、預測、攔阻（審計）。Cost Engine 永不決定「用誰」，Orchestrator 永不繞過計量。

## 6.2 條文

**C1（一切皆計量）** 每一次 Provider 調用——雲端或本地——必須計量。本地不是免費：至少計時間與運算佔用。**不存在不入帳的調用路徑。**

**C2（預算是硬約束）** 超出預算 → 停止並上報人工，**不得**靜默續行、不得「先跑完再說」。未設定預算時，生效的是保守預設預算，**不是**無限。

**C3（目標函數屬於使用者）** 路由在「預算、速度、精度」三者間的取捨順序由使用者政策決定。核心**不得**暗設偏好——特別是**不得**因任何商業關係、預設合作或作者喜好而偏向特定 Provider。路由理由永遠可查。

**C4（升級鏈與邊際原則）** 標準升級鏈：快取 → 本地輕量 → 本地強力 → 雲端經濟 → 雲端旗艦 → 人工。每一次升級必須存在「預期邊際效益 > 邊際成本」的依據（來自校準與記憶）；預期效益不足時，合法出路只有兩條：交給人工，或如實接受現狀並標記。**「不惜成本自動跑到最好」不是合法策略，除非使用者明示。**

**C5（人工有價）** 人的審核時間必須以估計成本入帳。「交給人」是一筆被定價的路由決策，不是系統的免費出口。此條與 P4 互為表裡：為人的注意力記帳，才會珍惜它。

**C6（預測必可證偽）** 每筆成本與時間預估都必須與實際值配對留存；預測器以本專案歷史持續再校準。不可證偽的預估（只給數字不留對帳）違憲。

**C7（帳本恆真）** 成本帳本 append-only。任何資產必須可回答：「你的總養成成本是多少？」——一版資料集、一顆模型、一次匯出，皆可溯源加總。

**C8（快取是一等成本槓桿）** 見 O9。重複呼叫相同三元組而未走快取，屬於浪費使用者資源的缺陷。

# 第七章　Project Intelligence（Project Memory）

## 7.1 立場

是。每個專案必須擁有自己的長期記憶——但記憶被嚴格定義為「**工作的副產品**」，以防它膨脹成一個新功能面。

## 7.2 條文

**M1（副產品原則）** 記憶由既有事件自動沉澱（審核行為、試鏡結果、決策結果、修正紀錄），**不得**要求使用者執行「餵記憶」的額外工作。使用者正常工作，記憶自然生長。

**M2（記憶屬於專案）** 記憶存放於專案資料夾內、隨專案攜帶、與專案共存亡。「這家公司的 Valve 指氣動閥」「本專案累計修正 800 次、六成集中在兩個混淆對」「Provider X 在『刮痕』類實測最強」——這些事實的家在專案裡，不在產品裡、更不在雲端。

**M3（跨專案禁運）** 記憶預設**不得**離開專案。跨專案聚合（「這位使用者的所有專案顯示……」）預設禁止；唯經使用者明示選擇加入，且僅允許匿名化聚合。此條是 P6 的延伸，位階同等。

**M4（條目有出處與時效）** 每條記憶必須記載出處（源自哪些事件）與有效條件。與 Provider 相關的記憶（成績、校準）在該 Provider 升版時自動失效待重驗。**過期的記憶比沒有記憶更危險。**

**M5（使用者對記憶的完全主權）** 可檢視、可修改、可刪除（遺忘權）。記憶不是黑盒人格，是一份使用者可以翻開的帳。

**M6（記憶是先驗，不是判決）** 記憶可影響路由排序與審核優先級，**不得**取代校準、試鏡與 Human Review。「Provider X 上次很準」永遠不構成「這次免審」的理由。

**M7（Schema 版本化）** 記憶結構必須版本化且附遷移路徑；十年間任何舊專案打開時，記憶必須可讀、可升級、不損毀。

# 第八章　2035 條款

## 8.1 永不淘汰（可終身重壓）

使用者主權與開放格式（P6/A7）；可驗證性與證據鏈（P5/A6）；人為最終權威與人工有價（A12/C5）；資料高於模型（P2/A4）；治理帳本（血統、版本、決策、成本）；反過度工程紀律（A10）；以及**本憲法的修憲程序本身**。

## 8.2 必然淘汰（設計時已判死刑，僅以可拋棄方式建造）

2026 年的每一個具名模型與 Provider；原始 confidence 的一切啟發式語意；以 token 計價的成本單位（故 Cost Engine 的計量單位必須可插拔）；當前的任務分類邊界（Detect/Segment/Pose 的區隔正在融解為統一感知）；「影像＝靜態 2D 圖」的預設（影片流、3D、多感測將常態化）；以及——誠實地說——**「訓練 Student」這件事本身的重要性可能下降**（邊緣端基礎模型持續變強變便宜）。本架構對此免疫：訓練只是 Provider 的一個可選動詞，若 2035 年蒸餾式微，被刪除的是一個動詞的使用頻率，不是核心的任何部分。**這正是架構正確的證明方式：任何單一趨勢的死亡都不傷及核心。**

## 8.3 現在必須預留的接口（F 條款——僅留契約，不建實作，受 A10 約束）

**F1（幾何可擴充）** Claim 的幾何欄位為型別化開放集（bbox／polygon／mask／keypoints／…），新幾何以新型別加入，不重寫 Schema。

**F2（模態可擴充）** 「輸入媒體」抽象不得綁死靜態影像；時間軸（影片段）、序列、多源輸入以媒體型別擴充容納。

**F3（計量單位可插拔）** Cost Engine 的「資源單位」是抽象的（金錢／token／秒／瓦），計價方式是掛載的。

**F4（能力詞彙 append-only）** 見 PR9。

**F5（任務即模組）** 見 A9。

**F6（Agent 可駕駛）** 2035 年的主要「使用者」很可能是另一個 AI Agent。核心必須完全可經 API 驅動（A3 的無頭原則即為此留門）；GUI 永遠只是眾多客戶端之一。

**F7（帳本序列化穩定）** Decision 與血統的儲存格式必須十年可讀：自描述、向後相容、附版本。2035 年重放 2026 年的決策，是本憲法對自己的驗收標準。

**F8（輸出契約穩定）** Application Pipeline 所依賴的 Label／Claim 輸出契約獨立版本化——下游業務系統不因核心升級而斷裂。

# 第九章　MVP Boundary

## 9.1 憲法級的一刀

**B1（可以晚做聰明，不可以晚做誠實）** 第一版的 Orchestrator 允許極簡——試鏡、校準信心分流、手動選擇 Provider，僅此而已。但 **Decision、成本、血統、版本的記錄格式必須從第一天完整**。理由：聰明的路由策略可以隨時升級，缺頁的帳本永遠補不回來——事後補記帳等於資產斷代。此條為 MVP 的最高指導原則。

## 9.2 第一版絕對不得做（做了即違憲）

**B2** 超過三個 Provider（一個雲端 VLM 老師、一個本地開放詞彙老師、一個可訓練學生——足夠滿足 Rule of Two 並暴露真實介面皺褶）。

**B3** 對外 Plugin API 與任何 marketplace（介面未經內部淬煉即對外，等於把技術債出口給社群）。

**B4** 自動多 Provider 表決、全自動成本最佳化路由（無累積數據時的「聰明策略」是隨機數生成器加自信）。

**B5** LLM Copilot（規則層先行；LLM 顧問屬 userland，其地基——可查詢的記憶與帳本——恰好正是 B1 要求先建的東西）。

**B6** Application Pipeline 建構器（僅履行 F8：輸出契約穩定文件化）。

**B7** 雲端訓練、多 GPU、多人協作、Node Editor、跨專案智慧。

## 9.3 現在做必然失敗清單（附死因）

一步到位的 Provider 大一統介面（死因：違反 A10，皺褶只能從實作中學）；聰明路由（死因：無歷史數據，違反 C6 的可證偽要求）；2035 模態支援（死因：需求尚無形狀，違反 A10）；完美的記憶本體論（死因：記憶分類學必須從真實審核行為中長出，先驗設計必錯）。

## 9.4 範圍的唯一判準

第一版的每一個功能必須通過此問：**「它是否服務『十分鐘內讓使用者親眼看到自己的圖被看懂，且每一步都在為他累積帶得走的資產』？」** 不服務者，一律出局。（功能內容以 R2 第 10.6 節 M0/M1 為準，本憲法不重列——重列即發散。）

# 第十章　Development Constitution

以下條文約束未來一切開發者、架構師與 **AI Agent**。凡對本倉庫的任何變更，變更說明必須能回答：「本變更未違反哪些 D 條文？若違反，修憲案在哪？」

**D1（核心不識 Provider）** 核心程式碼不得 import、連結或字串硬編碼任何具體 Provider。以自動化依賴檢查強制。

**D2（熱插拔）** 新增／移除 Provider 不得要求修改或重編核心。驗收方式：一個全新 Provider 套件投放後，無核心變更即可被發現、試鏡、路由。

**D3（Decision 可重放）** 任何自動決策必須記錄至可重放的程度：輸入參照、政策版本、候選集、選擇理由、Provider 版本、輸出。重放時使用**已記錄的輸出**，不重新調用機率性元件。

**D4（Cost 可追蹤）** 不存在未計量的調用路徑。新增任何呼叫 Provider 的程式路徑而未接 Cost Engine，即為缺陷。

**D5（Dataset 可回滾）** 一切資料變更經由版本化交易；禁止就地破壞性修改。任何歷史版本永遠可恢復。

**D6（AI 決策留證據）** 任何由模型參與的決策（分流、建議、診斷）必須保留 Evidence 並可由人複核。

**D7（Claim 不成 Label）** 未經 Human Review 的 Claim 永不進入 Dataset。此規則不存在任何模式、任何旗標、任何管理員權限可以關閉。

**D8（黃金三鐵律）** Golden Dataset 永不參與訓練、永不被機器覆寫、永為最終裁決。程式層面必須使違反此條在型別／權限上不可表達，而非僅靠紀律。

**D9（Schema 必附遷移）** 任何持久化結構的變更必須附自動遷移；任何歷史版本的專案資料夾必須永遠打得開。

**D10（破壞性操作可逆）** 一切破壞性操作：可逆，或二次確認＋完整留痕，二者必居其一。

**D11（LLM 不入帳房）** 記帳、版本、裁決路徑為確定性保留區，LLM 及一切機率性元件不得進入。

**D12（Rule of Two）** 抽象介面須有兩個真實實作方可固化；單實作抽象標記為暫定並記入 ADR。

**D13（依賴單向）** 介面層 → 服務 → Orchestrator → Provider，違反方向的 import 由 CI 拒絕。

**D14（錯誤雙層）** 一切面向使用者的錯誤必須同時具備：人話說明（發生什麼、下一步）與可展開的技術詳情。只有 stack trace 的錯誤是缺陷。

**D15（離線核心）** 核心工作流不得存在網路硬依賴。CI 中必須存在斷網測試場景。

**D16（格式開放）** 專案磁碟格式的文件與其實作同步維護；文件過期視同缺陷。

**D17（隱私即程式）** 任何將使用者資料送離本機的路徑，必須經過同一個顯式同意閘（per-project 雲端開關），且送出內容可審視。繞過同意閘的捷徑，不論動機，一律違憲。

**D18（Provider 合規套件）** 每個 Provider 必須通過 Conformance Suite 方可註冊；套件本身隨 Claim Schema 版本演進。

**D19（憲法優先）** 任何設計討論中，本憲法條文可被引用為終局依據；認為條文錯誤的唯一救濟是提出修憲案（ADR＋升版），不得以「特殊情況」擱置條文。

**D20（AI Agent 條款）** 任何 AI Agent 修改本倉庫前必須讀取本憲法；其提交說明必須聲明所觸及的憲法條文及合規性。Agent 與人類開發者在本憲法前完全平等——沒有任何一方可以「更懂所以例外」。
