## 一、分工的三條軸線（不是按「難度」分）

**軸線 1：可規格化程度。** 能事先寫死「輸入、輸出、驗收測試」的工作 → Codex，**再複雜也給它**（複雜演算法只要有 property test 框住就是安全的）。規格本身就是難題的工作（要做取捨、要定邊界）→ Claude。

**軸線 2：不可逆性。** 憲法 B1「可以晚做聰明，不可以晚做誠實」——帳本格式、Schema、專案磁碟格式這類**事後補不回來**的東西，錯一次就是資產斷代 → Claude。可以隨時重寫的東西（UI 頁面、轉換器、腳本）→ Codex。

**軸線 3：憲法分區。** 憲法已經畫好安全邊界，直接沿用為分工邊界：

| 憲法分區 | 內容 | 歸屬 |
|---|---|---|
| **確定性保留區**（A5/D11：帳、版本、裁決） | Decision/成本/血統帳本、Dataset 版本交易、閘門邏輯 | **Claude 專屬** |
| **核心契約區**（PR2：唯一耦合點） | Claim Schema、能力詞彙表、Provider 契約、遷移框架 | **Claude 設計**，Codex 可在凍結後實作周邊 |
| **Provider 圈**（PR1–PR9） | 各 adapter 實作 | **Codex**（Conformance Suite 當守門員） |
| **Userland**（GUI、管線、工具） | 頁面、匯入匯出、腳本、文件 | **Codex** 自由發揮 |

成本邏輯：Codex 吃掉約七至八成的 token 量（UI、adapter、測試、樣板），Claude 集中在兩三成但承擔八成風險的關鍵路徑。

## 二、模組歸屬表（M0 + M1）

### Claude 自己接手（不外包）

| 模組 | 為什麼必須是 Claude |
|---|---|
| Claim Schema ＋ 能力聲明詞彙表 | 全系統唯一耦合點（PR2），版本化策略錯了全盤重來；F1/F2 的擴充性要一次想對 |
| 專案資料夾格式、DB Schema、版本 manifest | A7 開放格式＋D9 永久可開，最不可逆的決策 |
| 血統／Decision／Cost 帳本資料模型 | B1 的「誠實」本體；append-only 與可重放（D3）的設計細節極易做錯 |
| Orchestrator v1（試鏡→校準→分流→降級狀態機） | 跨模組推理密集；O1–O10 的職責邊界要在程式碼裡守住 |
| 校準與統計引擎（信心校準、收縮估計、試鏡評分、統計閘門） | 統計判斷題不是實作題；小樣本收縮這種地方 Codex 會寫出「看起來對」的錯誤數學 |
| Golden Set 隔離機制設計 | D8 要求「型別上不可表達違規」，這是型別系統設計題 |
| 隱私同意閘（D17） | 安全邊界不外包 |
| Conformance Suite ＋ CI 守門（依賴方向 lint、core 禁 import、斷網測試） | **測試就是規格，規格就是設計**；這套測試同時是管 Codex 的韁繩 |
| 全部 ADR、憲法合規審查、觸及 /core 的 code review | D19/D20 的執法者 |

### Codex 負責（規格票驅動）

| 模組 | 交給它的前提 |
|---|---|
| Provider adapters ×3（雲端 VLM、本地開放詞彙、Ultralytics 學生） | 契約凍結＋Conformance Suite 綠燈即收貨 |
| GUI 全部（Electron 殼、審核中心、聚類網格、畫布、儀表板） | Bridge 介面由 Claude 凍結；UI 是 token 大戶且完全可重寫 |
| 匯入管線 Worker（解碼、EXIF 落實、雜湊、縮圖、pHash） | 演算法成熟、golden-file 測試齊備 |
| Embedding 服務整合 | Claude 選模型與定介面，Codex 接 |
| 格式轉換器（YOLO/COCO/VOC 匯入匯出） | 規格完備的紙上作業，附標準樣本測試 |
| 審核佇列／專案 CRUD、API endpoints | OpenAPI spec 先行 |
| Job/Worker 程序骨架 | 程序模型（隔離、取消語意）Claude 先定，管線程式 Codex 鋪 |
| 單元測試充實、測試資料產生器 | Claude 給測試意圖清單，Codex 展開 |
| 打包、安裝檔、CI 腳本、錯誤訊息目錄（D14 雙層）、文件、i18n | 疊代型粗活，便宜模型的主場 |

### 先後依賴（第一週的正確順序）

```
Claude：repo 骨架 → core 契約（Schema/介面）→ Conformance/property 測試 → CI 守門
   ↓ 契約凍結後
Codex（可平行開三線）：adapter 實作 ｜ 匯入管線 ｜ GUI 殼與頁面
   ↓
Claude：批次審查觸及 /core 的 PR；統計引擎與 Orchestrator 並行推進
```

**Claude 不動工，Codex 就沒有安全的施工圖。契約與測試先行，是整個協作的前提。**

## 三、交接協定

**1. 規格票制。** 每個 Codex 任務都是一張規格票，缺欄不發包：

```
【目標】一句話
【範圍】允許改動的檔案/目錄白名單
【介面】已凍結的簽名（不得擅改，有意見回報不動手）
【驗收】測試清單（先寫好，綠燈=完成的唯一定義）
【審查層級】L0／L1／L2（v1.1 新增，定義見 §3-2）
【憲法】本票觸及的條文（如 D13、D14）
【禁區】明列不得碰的目錄與行為
```

**2. 合併規則。** 依 §3-2 審查分級制執行。審查採批次制，不逐 PR 消耗。

### §3-2 審查分級制（v1.1 新增）

工作階級修正：**Claude 是稽核者，不是流水線上的品管員。** 逐票精讀是誤用，分級如下：

| 層級 | 適用 | 流程 |
|---|---|---|
| **L0 免審** | 純 userland 白名單路徑（ui/、providers/、app/ 的票面範圍內） | CI 全綠＋PR 含 D20 聲明與驗收勾選 → 人直接合併。**這是預設層級** |
| **L1 抽審** | 從已合併的 L0 票中抽樣 | Claude 事後抽查：起始每 3 張抽 1，連續無問題降至每 5 張抽 1；抽中發現問題 → 該承包者回到全審重新累積信任。（與產品內盲審抽樣＋信任衰減同一哲學，R2 9.3） |
| **L2 必審** | 觸及 core/、scripts/、.github/、docs/00-法規、docs/01-定義、任何已凍結介面或契約的變更；或票面明標 L2 | Claude 合併前審查，聚焦守門檔 diff 與介面一致性，**不逐檔精讀 userland 程式碼** |

**新承包者條款**：任何新 agent／模型／人的**首張票一律全審**（試鏡，與 PR5 同哲學）——票-0001 屬此類，下不為例。

**機械化預告**：「PR 只動票面白名單」的範圍檢查應由 CI 腳本承擔（scripts/，Claude 地盤）；依 A10 Rule of Two，累積三張票後實作，不預建。

**3. 鎖區制。** 同一模組同一時間只有一個 agent 在改。Claude 與 Codex 並行時以目錄劃界，避免合併地獄。

**4. D20 落地。** 兩個 agent 一視同仁：PR 說明必須聲明觸及的憲法條文與合規性。Codex 的 system prompt 應內嵌憲法第十章全文。

## 四、紅線（Codex 不得，CI 強制）

1. 不得修改 `/core` 的契約、Schema、帳本寫入路徑（CODEOWNERS ＋分支保護）。
2. 不得新增違反依賴方向的 import（D13 lint 直接擋）。
3. 不得碰 Golden Set 隔離、同意閘、版本交易的實作。
4. 不得「順手重構」票面範圍外的程式碼——Codex 的經典失敗模式是熱心越界。
5. 不得在規格模糊時自行發明 API——規格票缺欄就退票，不腦補。
6. 無測試不合併，測試紅燈不合併，**修改測試以讓它變綠視同造假**（測試是 Claude 定的規格）。

## 五、兩個 agent 的已知失敗模式（互相防守）

| 失敗模式 | 誰的毛病 | 防守 |
|---|---|---|
| 規格模糊時自信地發明介面 | Codex | 規格票缺欄退票；介面凍結制 |
| 熱心重構未委託的區域 | Codex | 白名單＋diff 範圍檢查 |
| 統計/數值上「看起來對」的錯誤 | Codex | 統計引擎不外包；property test 由 Claude 寫 |
| 過度設計、抽象太早 | Claude | 憲法 A10/D12（Rule of Two）反過來管 Claude 自己 |
| 審查瓶頸拖慢 Codex 產線 | Claude | §3-2 審查分級制；L0 免審自併 |
| 把稽核者當品管員用（逐票精讀） | 制度 | §3-2：L0 預設免審、L1 抽審、L2 才必審 |

---

## 六、一句話總結

> **Claude 蓋憲法管得到的地方（契約、帳本、裁決、統計），Codex 蓋憲法管不到的地方（介面、管線、外掛、粗活）；測試是兩者之間唯一的語言。** 規格票發得越乾淨，Codex 就越便宜好用；規格票發不出來的地方，就是 Claude 該自己動手的證據。
