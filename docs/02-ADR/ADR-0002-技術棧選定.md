# ADR-0002：選定 M0/M1 技術棧（殼層、前端、服務、儲存、守門工具鏈）

| 項目 | 內容 |
|---|---|
| 編號 | ADR-0002 |
| 日期 | 2026-07-07 |
| 狀態 | 🟩 **已採納**（李宗鴻於 2026-07-07 拍板：「我都沒意見」） |
| 決策者 | 李宗鴻（Claude 起草） |

## 背景
repo 骨架與 CI 守門（D1/D13 的自動化強制）無法在技術棧未定前動工；第一張 Codex 規格票也依賴凍結的工具鏈。R1 §5.2 與 R2 已論證方向（Web 殼＋Python 引擎），本 ADR 把方向落成具體選型。

## 決策
| 層 | 選型 | 一句話理由 |
|---|---|---|
| GUI 殼 | **Electron** | 捆綁 GB 級 Python runtime 後，殼層體積差異無關痛癢；開發效率與生態成熟度勝出（R2 5.2 論證） |
| 桌面能力隔離 | 自訂 **Bridge 介面層** | 前端不得直接碰 Electron API（A3）；未來可換 Tauri 而前端零改動 |
| 前端 | **React + TypeScript + Vite**，狀態管理 Zustand | 主流、Codex 產出品質最穩定的組合；輕量狀態管理夠用 |
| 本機服務 | **Python 3.12 + FastAPI**（HTTP＋WebSocket 事件流） | CV 生態在 Python；FastAPI 的 OpenAPI 自動生成正好當 Codex 規格票的介面附件 |
| Worker | 獨立 Python process（spawn，不用 thread） | 崩潰隔離與取消語意（R1 §5.3，憲法 PR4） |
| 儲存 | **SQLite**（全域 registry＋每專案 project.db）＋內容雜湊檔案區 | 單機零運維；SQLite 是 A7 認可的開放格式 |
| 契約載體 | **Pydantic v2** 定義 Claim Schema → 匯出 **JSON Schema** → TS 側 codegen | 單一事實來源，跨語言契約自動同步（PR2 的實作形態） |
| Python 環境 | **uv** | 快、可鎖定、可重現 |
| CI | **GitHub Actions**：import-linter（D13 依賴方向）＋核心禁 import 清單（D1）＋ruff＋pytest＋**hypothesis**（property tests）＋vitest＋dependency-cruiser（前端）＋斷網測試場景（D15） | 紅線全部機器強制，不靠自覺 |
| 打包 | electron-builder；Python runtime 以 python-build-standalone 預組（含離線完整安裝檔變體） | R2 9.5 的離線工廠場景是驗收條件 |

## 曾考慮的替代方案
- **Tauri 2** → 棄：R2 5.2 已論證；Bridge 層保留了日後遷移權。
- **PySide6 全 Python** → 棄：使用者於 R1 階段已選 Web 殼方向；且 Codex 對 React 生態的產出品質優於 Qt。
- **PostgreSQL** → 棄：桌面單機產品，SQLite 足夠且更符合專案自包含（P6）。
- **gRPC 取代 HTTP/WS** → 棄：本機通訊性能非瓶頸，OpenAPI 的規格票價值更高；Rule of Two 不支持現在就抽象傳輸層。

## 影響範圍
repo 骨架目錄佈局、CI 設定、分工協議中全部 Codex 票的技術前提。

## 憲法檢核（D19／D20）
- 觸及：A2/A3（Bridge＋依賴方向，由 import-linter 與 dependency-cruiser 強制）、A5（Pydantic 確定性驗證）、A7（SQLite＋JSON＋檔案，全開放格式）、A10（未預建多餘抽象；傳輸層、ORM 均未引入）、D1/D13/D15（CI 條目一一對應）、PR2/PR4。
- 合規聲明：無牴觸。本 ADR 不觸碰任何 Provider 選擇（那是 PR5 試鏡制的事，不入技術棧）。
