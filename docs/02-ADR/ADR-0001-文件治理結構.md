# ADR-0001：建立 docs/ 文件治理結構（分類、狀態制、登記制）

## 背景
專案已產生四份大型文件（R1、R2、憲法、分工協議），且開發期間必然持續產生大量討論與決策 md。若無分類、效力層級與完成度標記，三個月後將無人（包括 AI Agent）能分辨哪份文件還算數——這對「以文件為法」的開發模式是致命的。

## 決策
採用六分類目錄（00-法規／01-定義／02-ADR／03-規劃／04-評審／05-討論）＋五態狀態制（🟩定案／🟨討論中／🟦草稿／🟧部分失效／⬛已歸檔）＋三條登記義務（新文件登記索引、新名詞登記定義表、狀態變更同步）。討論檔經「畢業」機制落地為修憲案／ADR／規劃，永不直接成為法規。

## 理由
效力層級是憲法前言已定的秩序，目錄結構只是把它變成肉眼可見；狀態制直接回答「做完 vs 沒做完」；定義表登記制防止語彙漂移（多 Agent 協作時尤其致命）；ADR 系統是憲法修憲程序與 D19 的載體。

## 曾考慮的替代方案
- 單一資料夾＋檔名前綴 → 棄：規模化後不可讀。
- Wiki／外部工具 → 棄：違反 A7 開放格式精神（資產必須活在 repo 內、離線可讀）。
- 重新命名既有檔案以統一格式 → 棄：文件間存在大量互相引用的檔名，改名破壞引用；只遷移不改名。

## 影響範圍
四份既有文件遷移並加狀態標記；docs/README 成為所有進場者（含 Agent）的第一頁。

## 憲法檢核（D19／D20）
- 觸及條文：D19（憲法優先）、D20（Agent 必讀與合規聲明）、A7（開放格式精神）、修憲程序（前言）。
- 合規聲明：本結構是上述條文的執行機制，不新增效力、不修改任何條文；「畢業」機制確保討論不能繞過修憲程序影響法規。
