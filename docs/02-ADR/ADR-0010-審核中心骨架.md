## 背景

R2 定「審核中心」為主戰武器——「未來所有標註都是審核」，且是 M0 兩大不確定之一（另一是老師品質）。憲法 D7：**Human Review 是 Claim→Label 的唯一合法通道**。契約層已備齊：`Claim.review` 狀態機（`ReviewStatus`）、`Label`、`ReviewEvent`、`GoldenSetEntry` 與對應 append-only repo。缺的是把它們串成最小可跑閉環：**跑老師 → 存 → 人審 → Label → 黃金集 → 校準 → 信賴度亮起**。fixture Claim 已足以驗證審核 UX，不必等真老師。

讀契約逼出兩個前置相依：
- `Label.run_ref` 必填 → 被審的 Claim 必須來自**已存的 InferenceRun**（目前 `/infer` 無狀態、不存）。
- `Label.final_concept.taxonomy_node_id` 不得為 None → 需要**最小 Taxonomy**（概念→節點映射）才能生 Label。

## 決策

做**最小可跑閉環**，紀律性延後花俏部分（A10）：

1. **Run 記錄（最小 Orchestrator）**：新增有狀態的「process media」路徑＝跑 provider → 記一筆 `InferenceRun`（附最小 `DecisionRecord`／`CostEntry`，即使成本為 0）→ 存 `Claim` 序列（`RunRepository`）。既有無狀態 `/infer` 保留為即時預覽。
2. **最小 Taxonomy（概念登記）**：每專案一份 `raw_text → taxonomy_node_id` 登記；審核時概念若未映射，就地建節點並附 `ConceptMappingProvenance(kind="human")`。豐富版（概念卡／正反例／prompt 資產）延後。
3. **審核狀態機（服務層強制 D7）**：對已存 Claim——`approve`→（確認概念映射後）產 `Label`＋`ReviewEvent`；`edit`→調整幾何/概念產 `edited_approved` Label＋`ReviewEvent`；`reject`→僅 `ReviewEvent`。契約不變量已擋（批准必回指 label_ref）。
4. **聚類＝先按概念**：審核佇列按 `concept.raw_text` 分組批次審。**視覺相似聚類需 embedding 基礎設施，延後**（A10，不預建）。
5. **黃金集升級**：把一個 `Label` 標為 `GoldenSetEntry`（人選定基準；四職能之一）。
6. **飛輪閉合**：黃金集 Label → `calibrate`（ADR-0007）→ `CalibrationSnapshot` → `apply_calibration` 回填未來 Claim 的 `calibrated`／`reliability`。**這一步讓「看懂」站的信賴度三色從全灰真正亮起。**
7. **延後清單（紀律，各自另立 ADR/票）**：視覺相似聚類、盲審＋蜜罐＋衰減函數（R2 9.3）、豐富 Taxonomy、成本估計/預算執行、進階點擊生框編輯、主動學習排序。

UI：「整理 Curate」站落審核網格（按概念分組、approve/edit/reject、reliability 標示）。

## 理由

- 直取憲法 D7 與 R2「審核即主戰武器」；用**已存在的契約與 repo**串閉環，新增的是服務層狀態機與最小前置，不動凍結契約。
- **最小閉環先跑**：能用 fixture Claim 驗證審核 UX（R2 兩大不確定之一），且閉合飛輪讓校準/信賴度真正有輸入——比先做真老師更早驗證「這個迴圈轉不轉得動」。
- **A10 守紀律**：視覺聚類、盲審、豐富 taxonomy 都是真實使用後才知道怎麼做對的東西，先延後。

## 曾考慮的替代方案

- **先做真老師再做審核** → 棄：審核 UX 是同等不確定，且沒有審核就沒有 Label／黃金集／校準輸入，飛輪空轉。
- **視覺相似聚類一步到位** → 棄：需 embedding 基礎設施，屬 A10 不預建；先按概念分組已可驗證 UX。
- **跳過 run 記錄、直接審預覽 Claim** → 棄：`Label.run_ref` 必填、血統要求（P4/D5），無 run 即無合法 Label。
- **豐富 Taxonomy 先行** → 棄：Label 只需 `taxonomy_node_id` 存在；最小登記已足，概念卡延後。

## 影響範圍

- **新增（core/service）**：最小 run 記錄（InferenceRun＋Decision/Cost）、最小 Taxonomy 登記、審核狀態機服務（Claim→Label/ReviewEvent）、黃金集升級、校準回填 wiring。
- **消費既有**：`Claim`/`Label`/`ReviewEvent`/`GoldenSetEntry` 契約與 repo、校準引擎（ADR-0007）、provider 抽象（ADR-0008）、API/UI 橋（ADR-0009）。
- **UI**：新「整理 Curate」站；API 加審核相關端點。
- **相關 ADR**：ADR-0003（Claim/Label 唯一事實來源）、0004（帳本）、0007（校準）、0008（provider）、0009（橋）。屬組裝，無牴觸。

## 憲法檢核（D19／D20）

- **觸及條文**：D7（Claim→Label 唯一通道＝Human Review）、D8（黃金集三鐵律）、D5（append-only）、P4（人為判斷留痕＝ReviewEvent）、A10/D12（延後視覺聚類/盲審/豐富 taxonomy，不預建）、§3（Confidence/校準）、R2 §9.4/10.1（審核中心）。
- **合規聲明**：未修改憲法或凍結契約；審核狀態機在服務層強制 D7（契約不變量已機械擋批准必產 Label）。最小 Taxonomy 附映射血統（ConceptMappingProvenance）。延後項以 A10 明列，不預建抽象。
- **未牴觸**，無需修憲。惟採納後若新增跨文件結構名（如「處理/ProcessRun」「概念登記/ConceptRegistry」）→ 依例登記 01-定義（需人）。
