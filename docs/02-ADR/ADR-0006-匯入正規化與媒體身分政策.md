## 背景

匯入管線（票-0003）要把影像位元組落成 `MediaRecord`＋blob。ADR-0005 定了「內容雜湊 blob 庫」與 `MediaRecord.exif_normalized` 旗標，但**沒定兩件不可逆的事**：

1. **`media_hash` 到底雜湊哪一份位元組**——原始檔，還是 EXIF 轉正後的檔？`media_hash` 是媒體的永久身分，Claim／Label／Manifest／Golden 全部以它為錨（憲法第三章、D5）。一旦有專案落地就補不回、改不動（B1／D9）。
2. **由哪個影像庫做解碼與 EXIF 轉正**——不同庫的 transpose 行為會產生不同像素，污染身分雜湊。

規格模糊處若讓 Builder 自行發明，等於把不可逆決定外包給施工者（協議軸線 2、失敗模式表第一列）。故先立此 ADR 凍結政策，票-0003 引用之。

## 決策

### 1. 媒體身分 = 入庫位元組的 SHA-256

`MediaRecord.media_hash` ≡ `sha256(存入 blob 庫的位元組)` ≡ `MediaBlobStore.put(...)` 之回傳值。blob 庫以內容雜湊定址、`find()` 靠此雜湊定位；**身分與被儲存物必須是同一份位元組的雜湊**，否則 blob 找不回。此為 ADR-0005 §3 的必然推論，於此明文化。

### 2. EXIF 正規化採條件式策略（Policy B）

匯入時讀 EXIF Orientation，分兩路：

- **Orientation 缺失或 = 1（像素已正立）**：**原始位元組原封入庫**，`media_hash = sha256(原檔)`。不重編碼、不失真；跨匯入去重最大化。
- **Orientation ∈ {2..8}（需旋轉／翻轉）**：以 Pillow `ImageOps.exif_transpose` 轉正 → 重編碼為**同格式** → 入庫，`media_hash = sha256(轉正後位元組)`。方向標記於重編碼後不復存在。

兩路 `exif_normalized` 皆為 `True`（顯示方向＝像素方向，R1 §7.1「標註框整批錯位」災難從源頭消滅）。`width_px`／`height_px` 一律取**轉正後**的顯示尺寸。

### 3. 影像庫釘 Pillow

解碼、EXIF 轉正、縮圖統一由 **Pillow**。理由：跨庫 transpose 行為差異會污染身分雜湊，釘單一庫確保正規化確定性（A5）。Pillow 只存在於 `app/` 服務層——**core 仍禁止 import PIL（D1，`check_forbidden_imports.py` 已強制）**。

### 4. 格式封閉集與範圍界線

只接受 `MediaFormat` 封閉集（`jpeg`／`png`／`webp`／`bmp`／`tiff`，見 `contracts/media.py`）。其餘格式（gif／heic／mpo…）→ `UnsupportedMediaFormatError`。**多幀輸入**（動圖／影片幀序列）本票一律 `MultiFrameMediaError` 拒絕。HEIC、影片、pHash 去重為**後續票**（F2 模態擴充留門，A10 不預建）。

### 5. 縮圖規格（衍生快取，不參與身分）

縮圖是**可從 blob 重生的衍生快取，不進入身分雜湊、不進帳本**。規格：`media/thumbs/{media_hash}.jpg`；最長邊 ≤ `thumbnail_max_edge`（預設 256px）、保持長寬比、JPEG q80、由已正規化影像生成（方向正確）；temp+rename 原子寫入；已存在則跳過。縮圖遺失可重生，不影響資料完整性。

### 6. 冪等與原子性

- **冪等**：同一內容重複匯入 → 偵測 `media_hash` 已在庫 → 回傳既有 `MediaRecord`（`deduplicated=True`），不重複 `add`（避免 `ConflictError`）、不重生縮圖。
- **失敗無副作用**：先解碼＋格式／幀數驗證，**通過後才**寫 blob → 生縮圖 → `media.add`。解碼或格式驗證失敗時帳本與 blob 零寫入。（極端情況下 `blob.put` 成功而 `media.add` 前崩潰 → 產生孤兒 blob；因內容定址且 `put` 冪等，孤兒無害，留待後續回收（GC）票處理，不構成資料損毀。）

## 曾考慮的替代方案

- **Policy A：一律轉正重編碼** → 棄。JPEG 重編碼失真（違 P2 資料完整性）＋跨 Pillow 版本雜湊漂移＋去重破裂。
- **不轉正、保留 EXIF 方向標記** → 棄。重蹈「標註框整批錯位」，憲法／R1 明點要消滅。
- **雜湊原始檔、但入庫轉正檔**（身分與儲存物脫鉤）→ 棄。`find()` 靠 `media_hash` 定位 blob，脫鉤即取不回，直接違反 ADR-0005 blob 庫設計。

**已知取捨**：Policy B 的「轉正分支」雜湊依賴 Pillow 版本，非位元決定性。可接受——去重是**專案內**性質（P6 自包含），專案內同一 Pillow 版本內確定性成立；跨專案／跨版本對「已轉正影像」的去重不是正確性要求。多數影像 orientation=1 走原檔入庫，不受影響。

## 影響範圍

票-0003 匯入管線實作；`visionforge_app` 服務包新建並接入 CI（`app` job、workspace member）；下游一切以 `media_hash` 為錨的資產（Claim／Label／Manifest／Golden）。

## 憲法檢核（D19／D20）

- **P2／A4**：原檔優先不重編碼，資料完整性凌駕便利。
- **A7／D16**：媒體本體為標準影像檔，開放格式、脫離本產品可讀。
- **D5**：媒體記錄 append-only、內容不可變。
- **A5**：正規化政策確定、影像庫釘版，確定性成立。
- **D14**：匯入錯誤為結構化例外型別，雙層訊息於 API/UI 邊界完成。
- **A10**：HEIC／影片／pHash／多尺寸縮圖一律不預建，留待需求成形。
- **D1**：Pillow 限於 `app/`，core 不碰。

合規聲明：無牴觸。

票-0003（Codex）實作 `import_media()`；HEIC／影片／pHash 各開後續票；縮圖若日後 UI 需多尺寸，另開票（F8 輸出契約穩定）。
