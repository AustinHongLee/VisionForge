# ADR-0005：儲存層設計（文件式 SQLite＋內容雜湊 blob＋append-only repositories）

## 背景
匯入管線（下一張 Codex 票）與一切服務都需要儲存層；D5/D8/D9 的「型別級落實」必須在此層兌現。

## 決策
1. **文件式儲存**：契約 JSON 為權威資料，關聯欄位只是查詢索引——Pydantic 契約與儲存格式單一事實來源，無雙 schema 漂移；每筆記錄自帶 schema_version（PR2 N/N-1 在讀取端處理）。Manifest entries 正規化分表屬效能手段，讀取時重組並**全量重驗**（損毀版本讀不出來，而非悄悄用下去）。
2. **append-only 介面**：Repository 故意沒有 update/delete 方法（D5/D8 靠介面寫不出來，不靠紀律）；重複身分＝ConflictError。
3. **內容雜湊 blob 庫**：同內容只存一份；temp+rename 原子寫入（blob 只有「不存在」與「完整」兩態）。
4. **遷移框架**（D9）：schema_migrations 表＋只增不改的遷移清單；遇到更新版本的專案明確拒絕（ProjectSchemaTooNewError），不得靜默損毀。
5. **專案自包含**（P6/A7）：project.json 戶口名簿（原子寫入）＋ project.db ＋ media/blobs；複製資料夾＝完整備份。
6. 新增 **MediaRecord 契約**（contracts/media.py）：媒體戶籍（尺寸/格式/來源/exif_normalized）。

## 曾考慮的替代方案
ORM（SQLAlchemy）→ 棄：A10，文件式下 ORM 無增值；全正規化雙 schema → 棄：漂移風險；每記錄一檔案（無 DB）→ 棄：查詢與交易能力不足。

## 影響範圍
匯入管線票、審核服務、Orchestrator、Cost Engine 全部以 `visionforge_core.storage` 為儲存入口。

## 憲法檢核（D19/D20）
D5/D8（介面級落實）、D9（遷移＋拒新）、P6/A7（自包含開放格式）、A5（讀取全量重驗）、A10（無 ORM 無投機抽象）。合規。
